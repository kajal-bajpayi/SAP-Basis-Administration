# 📘 SAP TCode: ST22 — ABAP Runtime Error Analysis (ABAP Dump Analysis)

---

## 📌 What is ST22?

**ST22** is an SAP transaction code used to analyze **ABAP runtime errors** (commonly called **ABAP dumps** or **short dumps**). Whenever an ABAP program terminates abnormally during execution, SAP generates a short dump that is logged and can be reviewed through ST22.

It is one of the **most frequently used TCodes by SAP Basis Administrators** for troubleshooting application-level errors.

---

| **Default Retention** | Short dumps are kept for **2 days** by default |
| **Authorization Object** | `S_TCODE`, `S_ADMI_FCD` |

---

## 🧠 How ST22 Works

1. An ABAP program encounters a **runtime error** (e.g., division by zero, type conflict, memory overflow).
2. SAP kernel **terminates the program** and generates a **short dump**.
3. The dump is stored in the **SNAP table** in the database.
4. The Basis Admin or Developer opens **ST22** to view, analyze, and troubleshoot the error.

---

## 📂 Information Available in a Short Dump

A short dump in ST22 provides the following sections:

- **Error Type / Runtime Error Name** — e.g., `DBIF_RSQL_SQL_ERROR`, `TSV_TNEW_PAGE_ALLOC_FAILED`
- **Date and Time** of the error
- **User Name** who triggered the dump
- **Client and Server** details
- **Program Name and Line Number** where the error occurred
- **Error Analysis** — SAP's description of what went wrong
- **Source Code Extract** — the exact ABAP code section causing the error
- **Call Stack / How to Correct** — suggestions to fix the issue
- **System Environment** — OS, DB, kernel version, memory details

## 🔥 Common ABAP Runtime Errors (Short Dump Types)

| Runtime Error | Description |
|---|---|
| `TSV_TNEW_PAGE_ALLOC_FAILED` | Program ran out of memory (heap/stack overflow) |
| `DBIF_RSQL_SQL_ERROR` | SQL-level database error |
| `TIME_OUT` | Program exceeded maximum runtime |
| `RABAX_CALLING_ASSERT` | Assertion failure in ABAP code |
| `MESSAGE_TYPE_X` | Program raised an X-type (termination) message |
| `SYSTEM_NO_ROLL` | Roll area (user context memory) exhausted |
| `LOAD_PROGRAM_NOT_FOUND` | Required program/include not found |
| `CONVT_NO_NUMBER` | Type conversion error (string to number) |
| `COMPUTE_INT_ZERODIVIDE` | Division by zero |

## 🛠️ Key Actions a Basis Admin Performs with ST22

1. **Daily Monitoring** — Check for recurring or critical dumps.
2. **Identify Root Cause** — Read the error analysis and source code extract.
3. **Assign to Developer** — Forward relevant dumps to ABAP developers for code fix.
4. **Memory Issues** — If `TSV_TNEW_PAGE_ALLOC_FAILED`, check and increase memory parameters (`abap/heap_area_total`, `ztta/roll_extension`).
5. **Timeout Issues** — If `TIME_OUT`, check `rdisp/max_wprun_time` parameter.
6. **Database Errors** — If `DBIF_RSQL_SQL_ERROR`, coordinate with DBA team.
7. **Housekeeping** — Manage SNAP table growth; adjust retention via `rsnapshot/cleanup_days`.

## ⚙️ Important SAP Parameters Related to ST22

| Parameter | Purpose | Default |
|---|---|---|
| `rsnapshot/cleanup_days` | Number of days to retain short dumps | 2 days |
| `rdisp/max_wprun_time` | Max runtime for a dialog work process (seconds) | 600 |
| `abap/heap_area_total` | Total heap memory for all work processes | System-dependent |
| `abap/heap_area_dia` | Heap memory for a single dialog WP | System-dependent |
| `ztta/roll_extension` | Extended memory per user session | System-dependent |

## 🎯 Interview Questions & Answers

### Q1: What is ST22 used for?
> ST22 is used to analyze ABAP runtime errors (short dumps). It shows details like the error type, the program and line number that failed, the user, timestamp, and suggestions for correction.

### Q2: Where are short dumps stored?
> Short dumps are stored in the **SNAP** database table.

### Q3: What is the default retention period for short dumps?
> By default, short dumps are retained for **2 days**. This can be changed using the parameter `rsnapshot/cleanup_days` or via report `RSSNAPDL`.

### Q4: A user reports a "TIME_OUT" dump. What do you check?
> - Check the parameter `rdisp/max_wprun_time` (default 600 seconds).
> - Analyze the program in ST22 to see which part is consuming time.
> - Check if the program has inefficient SELECT queries or infinite loops.
> - Coordinate with the ABAP developer for code optimization.

### Q5: What does `TSV_TNEW_PAGE_ALLOC_FAILED` mean?
> It means the program has exceeded the available memory (heap area). You should check memory parameters like `abap/heap_area_total`, `abap/heap_area_dia`, and `ztta/roll_extension`. If the parameters are adequate, the issue is likely in the ABAP code (loading too much data into internal tables).

### Q6: How do you delete old short dumps manually?
> Run the report **`RSSNAPDL`** or program **`RS_SNAP_CLEANUP`** to delete old short dumps from the SNAP table.

### Q7: Can you see short dumps from other application servers in ST22?
> Yes. ST22 shows dumps from **all application servers** in the SAP system since the data is stored centrally in the SNAP database table.

### Q8: What is the difference between ST22 and SM21?
> | ST22 | SM21 |
> |---|---|
> | Analyzes ABAP runtime errors (short dumps) | Analyzes system log entries |
> | Stored in SNAP table | Stored in log files on each app server |
> | Focuses on program-level failures | Covers broader system events (login failures, WP restarts, etc.) |

### Q9: How would you monitor ST22 proactively?
> - Use **Solution Manager** or **SAP Focused Run** for automated alerting.
> - Schedule a **CCMS monitor** to track short dump counts.
> - Set up **daily checks** as part of Basis daily monitoring checklist.
> - Use report **RSSNAPDL** to manage SNAP table size.

### Q10: A critical program is dumping repeatedly. What is your approach?
> 1. Open **ST22** and read the dump details (error type, program, line number).
> 2. Check if it's a known SAP issue by searching **SAP Notes** (via `SNOTE` or SAP Support Launchpad).
> 3. If memory-related — review and adjust memory parameters.
> 4. If code-related — share the dump details with the ABAP developer.
> 5. If database-related — involve the DBA team.
> 6. Apply the fix, test, and monitor to confirm the issue is resolved.
